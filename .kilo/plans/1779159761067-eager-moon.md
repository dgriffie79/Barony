# Plan: Strip & Simplify Barony for WASM Target

## Goal

Prepare the Barony codebase for eventual WASM deployment (no Emscripten) by:
1. Removing all non-Windows platform code
2. Removing proprietary SDK integrations (Steam, EOS, PlayFab)
3. Removing unnecessary optional dependencies (imgui, TheoraPlayer, cURL, Opus, Tremor)
4. Replacing RapidJSON with a small C JSON library
5. Converting C++ to pure C (keeping audio working throughout)

**Critical constraint: Audio must remain working at every stage.** The game uses FMOD (C++ API). We convert it to FMOD's C API to keep it working during C++→C conversion. Audio is the LAST thing replaced (with Web Audio API for WASM).

**Scope**: ~555K lines, 164 `.cpp` files, 57 `.hpp` files, 4 `.h` files (~500K lines game code + ~48K ImGui third-party)

---

## Audio Preservation Strategy

The audio system stays working through the entire migration:

| Stage | Audio State | How |
|---|---|---|
| Stripped C++ | Working | FMOD C++ API, same as today |
| C++→C conversion | Working | Audio subsystem converted from FMOD C++ API to FMOD C API |
| Pure C | Working | FMOD C API continues to work via `zig cc` |
| C→Zig replacement | Working | C audio code called from Zig via `@cImport` |
| WASM target | Working → replaced | FMOD C API stubbed; Web Audio API backend added |

**FMOD's C API** (`fmod.h`) is a first-class interface, not a wrapper — it provides the same functionality as the C++ API (`fmod.hpp`). The conversion is mechanical:
```cpp
// C++ API (current)
FMOD::System* system;
system->createSound(name, mode, 0, &sound);
system->playSound(sound, 0, false, &channel);

// C API (converted)
FMOD_SYSTEM* system;
FMOD_System_CreateSound(system, name, mode, 0, &sound);
FMOD_System_PlaySound(system, sound, 0, 0, &channel);
```

## Phase 2: Strip Platform Code

### 2.1 Remove non-Windows platform `#ifdef` blocks

~48 total instances across game code (plus 16 in ImGui third-party):

| Define | Files | Game Code Count |
|---|---|---|
| `#ifdef APPLE` | `game.cpp`, `init.cpp`, `editor.cpp`, `savepng.cpp`, `ui_general.cpp` | 15 |
| `#ifdef LINUX` | `main.cpp`, `game.cpp`, `editor.cpp`, `ui_general.cpp`, `eos.cpp`, `init_audio.cpp` | 12 |
| `#ifdef BSD` / `__FreeBSD__` / `__NetBSD__` | `game.cpp` | 3 |
| `#ifdef PANDORA` | `init.cpp`, `game.cpp` | 4 |
| `#ifdef HAIKU` | `game.cpp` | 1 |
| `#ifdef NINTENDO` | 22 files (mainly UI, init, input, audio) | 334 |
| `_MSC_VER` | `init.cpp` | 1 |

**Strategy**: Keep the `#else` / fallback path, remove the `#ifdef` branch. In many cases these guard OS-specific paths, clipboard APIs, window setup, and data directory resolution. The Windows paths are already the default in most cases — the `#ifdef` blocks add non-Windows alternatives.

Files with significant non-Windows code:
- `src/game.cpp` — macOS bundle path, Linux data directory, FreeBSD/NetBSD clock, Pandora clock
- `src/init.cpp` — macOS bundle path, Pandora keymapping, Linux clipboard
- `src/editor.cpp` — macOS cursor handling  
- `src/main.cpp` — Linux gameover handling
- `src/savepng.cpp` — macOS save path
- `src/interface/ui_general.cpp` — macOS clipboard access

### 2.2 Remove ImGui third-party `_WIN32`, `__APPLE__`, `__linux__`, `__EMSCRIPTEN__`, `__ANDROID__` checks

~15 instances in `src/imgui/`. Since ImGui will be removed entirely in Phase 5, these can be deferred or handled when imgui is removed.

## Phase 6: Replace RapidJSON with C JSON Library

### 6.1 Choose replacement

Options: **cJSON** (public domain, single .c + .h, ~4K lines) or **yyjson** (MIT, very fast, single .h). cJSON is simpler and more widely portable.

### 6.2 Replace the JSON abstraction layer

**`src/json.hpp`** (201 lines): The `FileInterface` abstract base class + `FileHelper` template facade. This is a C++ serialization framework with:
- Virtual dispatch (will need to convert to struct-of-function-pointers or remove the abstraction)
- Templates for `value<T>()` for vectors, pairs, enums, classes, fixed arrays
- `std::function` for the serialization callback
- `std::bind` in `writeObject()`/`readObject()`

**Strategy**: Replace the entire `FileInterface`/`FileHelper` abstraction with a direct C JSON approach. Instead of a virtual dispatch layer, the ~15 files that currently use `FileHelper::writeObject<T>()` / `FileHelper::readObject<T>()` will directly call C JSON functions for serialization.

**`src/json.cpp`** (535 lines): RapidJSON-based implementation.
- `JsonFileWriter` → replaced by cJSON creation + `cJSON_Print()` / `cJSON_PrintPrealllocated()`
- `JsonFileReader` → replaced by `cJSON_Parse()` with manual value extraction 
- `BinaryFileWriter` / `BinaryFileReader` → kept as-is (these don't use RapidJSON)

### 6.3 Replace direct RapidJSON usage

~60 `rapidjson::StringStream` + `rapidjson::Document` create/parse patterns across these files:

| File | Usage | Lines |
|---|---|---|
| `light.cpp` | Load `data/lights.json` | ~76 |
| `init.cpp` | Load `data/tiles.json` | ~36 |
| `book.cpp` | Read/write compiled books cache | ~356 |
| `mod_tools.hpp` | Custom monster/curve/gameplay managers | ~1,300 lines of JSON code |
| `mod_tools.cpp` | Seed names, IRC config, challenge scenarios | ~200+ |
| `scores.cpp` | Save/load scores with FileReadStreamCustomWrapper | ~800+ |
| `playfab.cpp` | Score update files | ~80 |
| `init_game.cpp` | Load achievements JSON | ~56 |
| `ui/MainMenu.cpp` | Various JSON data | ~4 |
| `ui/GameUI.cpp` | Various JSON data | ~5 |
| `interface/interface.cpp` | Various JSON data | ~2 |
| `interface/identify_and_appraise.cpp` | Various JSON data | ~1 |
| `interface/consolecommand.cpp` | Various JSON data | ~2 |

**Strategy**: Replace each pattern mechanically:
```c
// Before (C++ with RapidJSON):
rapidjson::StringStream is(buf);
rapidjson::Document d;
d.ParseStream(is);
int val = d["key"].GetInt();

// After (C with cJSON):
cJSON* doc = cJSON_Parse(buf);
cJSON* key = cJSON_GetObjectItem(doc, "key");
int val = key->valueint;
cJSON_Delete(doc);
```

### 6.4 Update `build.zig` for C JSON

Add the new C JSON source file to the compile list. Remove RapidJSON include path from dependency setup.

## Phase 7: C++ → C Language Conversion

### 7.1 Containers — Replace std:: library

| C++ Type | Count | C Replacement |
|---|---|---|
| `std::string` | 1,124 | `char*` + length struct, or a `String` type with buffer management |
| `std::vector` | 261 | Dynamic array struct: `{ T* data; size_t size; size_t capacity; }` |
| `std::map` | 97 | Hand-rolled sorted array or simple hash table (depends on usage pattern) |
| `std::pair` | 79 | Anonymous struct `{ T1 first; T2 second; }` |
| `std::unordered_map` | 18 | Simple hash table or sorted array |
| `std::set` | 17 | Sorted dynamic array |
| `std::list` | 7 | Doubly-linked list struct (or convert to vector) |
| `std::unordered_set` | 6 | Hash set |
| `std::queue` | 5 | Ring buffer or dynamic array |
| `std::array` | 3 | Plain C array |
| `std::lock_guard` / `std::mutex` | 5 | Platform mutex API (SDL_mutex etc.) |
| `std::max` / `std::min` | 1,892 | `#define MAX(a,b)` / `#define MIN(a,b)` macros |

### 7.2 Control flow — Replace C++-only constructs

| Construct | Count | C Replacement |
|---|---|---|
| `auto` variable init | 6,020 | Explicit type |
| Range-based `for (x : y)` | 605 | Index or pointer loop |
| Lambda expressions | 1,032 | Named function + struct for captures |
| `static_cast<T>(x)` | 936 | `(T)x` |
| `const_cast<T>(x)` | 114 | Fix const-incorrect interfaces; or `(T)x` |
| `reinterpret_cast<T>(x)` | 36 | `(T)x` |
| `new`/`delete` | 323 | `malloc(size)` / `free(ptr)` |
| `constexpr` | 104 | `#define` or `static const` |
| `const` member funcs | 117 | `const` parameter (e.g. `const MyStruct* this`) |
| Default parameters | 335 | Wrapper functions or varargs |

### 7.3 Types — Replace `enum class` and class/struct

| Construct | Count | C Replacement |
|---|---|---|
| `enum class` | 13 | Plain `enum` with prefix naming |
| `class` → `struct` | 152 | `struct` (no method access control) |
| `T::serialize(FileInterface*)` pattern | ~30+ | Free function `T_serialize(T*, cJSON*)` |
| Function overloading | Many | Unique function names (e.g. `draw_int()`, `draw_float()`) |
| Constructor/destructor | 31 | `T_init(T*)` / `T_deinit(T*)` functions |
| `= default` / `= delete` | 33 | Remove or replace with explicit definitions |

### 7.4 Remove C++ headers

Replace all `#include <*>`:
- `<string>` → remove or replace with custom string header
- `<vector>` → remove or replace with custom vector header
- `<map>` → remove or replace with custom map header
- `<algorithm>` → replace with hand-written loops
- `<functional>` → remove (no lambdas, no std::function)
- `<cassert>` → `<assert.h>`
- `<cmath>` → `<math.h>`
- `<cstdio>` → `<stdio.h>`
- `<cstdlib>` → `<stdlib.h>`
- `<cstring>` → `<string.h>`
- `<cstdint>` → `<stdint.h>`
- All ImGui includes → remove (Phase 5)
- All fmod includes → remove (Phase 4)
- All steam_api.h, eos_sdk.h, playfab includes → remove (Phase 3)

### 7.5 Convert FMOD C++ API to FMOD C API

The audio subsystem uses FMOD's C++ API (`fmod.hpp`) via classes like `FMOD::System`, `FMOD::Sound`, `FMOD::Channel`, `FMOD::ChannelGroup`, `FMOD::DSP`, `FMOD::Reverb`, `FMOD::SyncPoint`. These must be converted to the equivalent C API (`fmod.h`):

| C++ API | C API |
|---|---|
| `FMOD::System*` | `FMOD_SYSTEM*` |
| `system->createSound(...)` | `FMOD_System_CreateSound(system, ...)` |
| `system->playSound(...)` | `FMOD_System_PlaySound(system, ...)` |
| `system->update()` | `FMOD_System_Update(system)` |
| `channel->setVolume(...)` | `FMOD_Channel_SetVolume(channel, ...)` |
| `FMOD::Sound*` | `FMOD_SOUND*` |
| `FMOD::Channel*` | `FMOD_CHANNEL*` |
| `FMOD::ChannelGroup*` | `FMOD_CHANNELGROUP*` |
| `FMOD::DSP*` | `FMOD_DSP*` |

Files to convert:
- `src/engine/audio/sound.hpp` — type declarations, globals, VoiceChat_t, EnsembleSounds_t
- `src/engine/audio/defines.cpp` — pointer declarations
- `src/engine/audio/init_audio.cpp` — init/shutdown
- `src/engine/audio/sound.cpp` — sound_update, setGlobalVolume, error checking
- `src/engine/audio/sound_game.cpp` — playSound* variants, VoiceChat, EnsembleSounds
- `src/engine/audio/music.cpp` — playMusic, stopMusic, loadMusic

**Strategy**: Switch from `#include <fmod.hpp>` to `#include <fmod.h>`. Replace all `->` method calls with `FMOD_*` function calls. This is mechanical — every C++ method has a direct C equivalent.

**Verification**: Build with `zig c++` after conversion. Game should run with identical audio behavior.

### 7.6 ImGui-specific C++

~48K lines of C++ in `src/imgui/`. If kept, these must be isolated as a C++ module compiled separately and linked as a static library. If removed (Phase 5.1), no action needed.

## Phase 8: Rename to .c and Update Build

### 8.1 Rename files

Rename all `.cpp` → `.c` after all C++ features are removed.
Rename `.hpp` → `.h` or inline into `.c` files.

### 8.2 Update `build.zig`

- Change source file list from `.cpp` to `.c`
- Change compile flags: remove `-std=c++17`, use `-std=c99` or `-std=c11`
- Remove `link_libcpp = true` from module setup
- Remove C++ runtime linking
- Change module from `createModule` with `link_libcpp` to plain C module

### 8.3 Verify C compilation

After conversion, build with `zig build` targeting native Windows. Then test with:
```
zig build -Dtarget=x86_64-windows-gnu -Doptimize=ReleaseFast
```

## Execution Strategy

**Key principle: the game must build and run (with audio) at every stage.**

```
Strip SDKs → Strip platform code → Strip OpenAL/optional deps → 
→ Replace RapidJSON → Convert C++→C (keeping FMOD audio) → Rename .cpp→.c
                                                ↓
                                    FMOD C API keeps audio working
```

1. **Phase 1** (Config + build.zig) — trivial, game builds
2. **Phase 2** (Platform strip) — verify build after each file group (game runs)
3. **Phase 3** (SDK strip) — verify build after each SDK removed (game runs, no lobby/auth)
4. **Phase 4** (Strip OpenAL) — trivial, OpenAL was unused
5. **Phase 5** (Optional deps) — verify build after each removal (game runs, no imgui/video)
6. **Phase 6** (JSON replacement) — verify build (game runs, save/load tested)
7. **Phase 7** (C++ → C) — sub-phases, each verified with a build:
   - 7a. Replace `std::max`/`min` with macros
   - 7b. Replace `enum class` with plain enum
   - 7c. Replace `new`/`delete` with `malloc`/`free`
   - 7d. Replace `static_cast` etc. with C casts
   - 7e. Replace `constexpr` with `#define`
   - 7f. Replace `std::vector` with custom dynamic arrays
   - 7g. Replace `std::map`/`unordered_map` with custom tables
   - 7h. Replace `std::string` with `char*`
   - 7i. Replace lambdas with named functions
   - 7j. Replace range-for with index loops
   - 7k. Replace `auto` with explicit types
   - 7l. Convert classes to structs + free functions
   - 7m. Replace function overloading
   - 7n. Replace default parameters
   - 7o. Remove C++ headers
   - 7p. **Convert FMOD C++ API to C API** ← audio stays working
8. **Phase 8** (Rename .cpp → .c, switch to C compiler) — final step

**Audio-specific verification**: After Phase 7p and Phase 8, run the game and confirm:
- Sound effects play (attacks, UI, items, ambient)
- Music plays and transitions (explore, combat, boss)
- Volume controls work

## Risk Assessment

| Risk | Mitigation |
|---|---|
| Build breaks mid-Phase 7 | Each sub-phase is small; build-verify after each |
| std::string (1,124 uses) is tedious | Batch rename with script, then fix compilation errors |
| cJSON may not support all RapidJSON patterns | Test with actual game data files (scores.json, lights.json, etc.) |
| 1,032 lambda conversions are manual work | Each lambda is typically 1-5 lines; cluster by file |
| SDL2 windowing has no WASM equivalent yet | This is for the final WASM phase; for now we still target Windows native |
| PhysFS may be tricky for WASM VFS | Deferred — we replace PhysFS with a custom VFS only when targeting WASM |
