# Plan: Migrate Barony Build System from CMake to Zig

## Overview

Replace the existing CMake build system with Zig's build system (`build.zig`), using `zig cc`/`zig c++` as the compiler. The game (~580k lines C++17 across ~200 source files) produces two executables: `barony` (game) and `editor` (map editor).

This plan is **incremental** â€” first get a working compile with core dependencies, then layer on optional features.

---

## Step 1: Prerequisites Setup

1. **Zig 0.16.0** is installed and available at `zig`
2. **Install core system dependencies** via vcpkg at `C:\dev\vcpkg`:
   ```bash
   git clone https://github.com/Microsoft/vcpkg.git C:\dev\vcpkg
   cd C:\dev\vcpkg
   bootstrap-vcpkg.bat
   vcpkg install sdl2 sdl2-image sdl2-net sdl2-ttf physfs rapidjson libpng zlib --triplet x64-windows
   ```
3. `build.zig` will accept `-Dvcpkg_root="C:\dev\vcpkg"` to locate libraries automatically
4. Linux: `apt install libsdl2-dev libsdl2-image-dev libsdl2-net-dev libsdl2-ttf-dev libphysfs-dev rapidjson-dev libpng-dev libz-dev libgl-dev`
5. macOS: `brew install sdl2 sdl2_image sdl2_net sdl2_ttf physfs rapidjson libpng`

## Step 2: Create `build.zig`

Create `build.zig` in the repo root with the following structure:

### 2.1 Executable Targets

```zig
// Two executables:
// - barony (game)
// - editor (map editor, shared code + editor-specific files)
```

- `barony` target: all GAME_SOURCES (~159 .cpp files + ~7 imgui .cpp files + ~5 audio engine .cpp files)
- `editor` target: all EDITOR_SOURCES (~36 .cpp files, overlapping with game sources)

### 2.2 Source File Lists

Inline all source file paths organized by subdirectory (mirroring the CMakeLists.txt files):

- `src/*.cpp` â€” main source directory (~90 files)
- `src/magic/*.cpp` â€” 6 files
- `src/interface/*.cpp` â€” 16 files
- `src/ui/*.cpp` â€” 11 game + 9 editor files
- `src/engine/audio/*.cpp` â€” 5 game + 3 editor files
- `src/imgui/*.cpp` â€” 7 files

### 2.3 Compiler Configuration

```zig
const exe = b.addExecutable(.{ .name = "barony", .target = target, .optimize = optimize });
exe.linkLibCpp();

// Add all C++ source files with shared compile flags
const cxx_flags = &.{
    "-std=c++17",
    "-ffast-math",
    "-funroll-loops",
    "-fstrict-aliasing",
    "-Wno-pedantic",
    "-Wno-empty-body",
    "-Wno-string-plus-int",
    "-Wno-parentheses",
    "-Wno-format-security",
    "-Wno-multichar",
    "-Wno-inconsistent-missing-override",
};

// Add source files grouped by directory
exe.addCSourceFile(.{ .file = b.path("src/main.cpp"), .flags = cxx_flags });
// ... all source files, and:
// For directories: exe.addCSourceFiles(.{ .files = &..., .flags = cxx_flags });
// (exact API to be confirmed during implementation â€” may use addCSourceFiles with a struct)
```

### 2.4 Config.hpp Generation

We cannot use `@VAR@` substitution like CMake. Instead:

- **Option A**: Generate `Config.hpp` at build time via a Zig build step that reads `Config.hpp.in` and substitutes values based on build options. This is the cleanest approach.
- **Option B**: Pre-generate `Config.hpp` with defaults and check it in. Simpler but less flexible.
- **Option C**: Pass all config as compile-time `-D` flags and keep `Config.hpp` as a thin wrapper that `#ifndef`-guards each define. This avoids file generation entirely.

**Recommended: Option A** â€” Write a small Zig function that processes the template and writes `src/Config.hpp` before compilation.

### 2.7 Platform Detection

In `build.zig`, detect target OS and set platform-specific defines/libraries:

- **Windows**: `WINDOWS` define, `-lopengl32`, `-lws2_32`, `-lwsock32`, Win32 resource files for icons
- **macOS**: `APPLE` define, `-framework OpenGL`, `-framework IOKit`, `-stdlib=libc++`
- **Linux**: `LINUX` define, `-lGL`, `-lpthread`, `-rdynamic`, `-Wl,-rpath='$ORIGIN'`
- **BSD**: `BSD` define
- **Haiku**: `HAIKU` define

### 2.8 Include Paths

```zig
exe.root_module.addIncludePath(b.path("src"));
// Plus paths for external dependencies:
//   - RapidJSON headers: add system include path
//   - Optional: Steamworks, EOS, etc â€” add via -D or addIncludePath
```

For system dependencies, use `exe.root_module.linkSystemLibrary("SDL2", .{})` etc., which auto-resolves include paths for well-known system libraries.

For defines: `exe.root_module.addCMacro("WINDOWS", "")` or the equivalent per-target API.

### 2.9 Windows Resources

For Windows builds, embed `.ico` files:
- Use `windres` or the `.rc` file approach via zig cc's `-rc` support, or embed as a binary blob.

## Key Challenges & Mitigations

| Challenge | Mitigation |
|-----------|------------|
| `Config.hpp.in` substitution | Write a Zig build step that reads the template and substitutes `@VAR@` placeholders |
| C++17 standard library | `linkLibCpp()` links zig's bundled libc++ which supports C++17 |
| Win32 entry point (no `WinMain`) | SDL handles this via its `SDL_main` mechanism; standard `main()` should work |
| `.rc` files on Windows | Convert to compile as resource via zig cc, or embed icon manually |
| Optional dependency paths | Use `-D` options for library paths; require explicit paths for non-system SDKs (Steamworks, EOS) |
| Large number of source files | List all files explicitly in `build.zig` (grouped by directory for maintainability) |
| `#include "Config.hpp"` | Generated file must exist before compilation; run generation as a build step |
| Duplicate `main()` functions | Game and editor are separate executables with their own `main()` â€” keep them as separate build targets |

## Implementation Steps (Detailed)

### Step A: Write `build.zig.zon`
Create minimal manifest file.

### Step B: Write `build.zig` â€” Scaffold
1. `pub fn build(b: *std.Build) void` entry point
2. Define target and optimize
3. Accept `vcpkg_root` as a build option (string, default `null`)
4. If `vcpkg_root` is set, derive include/library paths:
   - `{vcpkg_root}/installed/x64-windows/include`
   - `{vcpkg_root}/installed/x64-windows/lib`
5. Create `barony` executable
6. Add all game source files (list from all subdirectories)
7. Add include paths: `src/` + vcpkg/system include paths
8. Add all C++ compile flags: `-std=c++17`, `-ffast-math`, `-funroll-loops`, etc.

### Step C: Write `build.zig` â€” Config.hpp generation
A function in `build.zig` that runs as a build step before compilation:
1. Reads `src/Config.hpp.in` as a string
2. Replaces `@Windows@`, `@Linux@`, `@Apple@` etc. with `1` or `0` based on `target.result.os.tag`
3. Replaces `@FMOD@`, `@STEAMWORKS_ENABLED@`, etc. based on build options
4. Replaces `@IMGUI@` with `1` (always enabled)
5. Writes the result to `src/Config.hpp`
6. This runs via `b.addWriteFiles()` or a custom step that produces the file, then we add it as a dependency of the compile step via `exe.step.dependOn(&gen_step.step)`

### Step D: Write `build.zig` â€” Linking
1. On Windows with vcpkg:
   - Set library search path to `{vcpkg_root}/installed/x64-windows/lib`
   - Link: `SDL2`, `SDL2_image`, `SDL2_net`, `SDL2_ttf`, `physfs`, `png`, `z`, `opengl32`, `ws2_32`, `wsock32`, `mingw32` (for SDL_main)
2. Use `exe.root_module.linkSystemLibrary("name", .{})` for each system lib
3. Use `exe.root_module.addLibraryPath(b.path("..."))` to add vcpkg lib dir
4. On macOS: link frameworks (`OpenGL`, `IOKit`)
5. On Linux: link `GL`, `pthread`, `dl`, `rt`

### Step E: Write `build.zig` â€” Editor target
1. Create `editor` executable target
2. Add editor sources (subset of game files + editor-specific files)
3. Add `-DEDITOR` compile flag
4. Same linking as barony

### Step F: Verification
1. `zig build` â€” compile barony (and editor by default)
2. If missing dependencies, provide clear instructions on how to acquire them
3. Test with `zig build run` (or running the binary directly with assets)
